OpenRouter Integration — Design Plan

FinTool · personal finance web UI · drafted 2026-06-25
Goal: alternative provider Curated list + free-text Local heuristic cost Status: design — not yet built

Summary

Add OpenRouter as a second chat provider alongside the existing Anthropic-direct path — the current behavior stays the default and untouched. A new provider abstraction isolates the one genuinely divergent concern (the API wire format) behind a small testable interface, so chat.js stops hardcoding Anthropic specifics.

Confirmed choices: alternative provider (not a replacement) · curated model list + free-text slug override for OpenRouter · local chars/4 × price cost heuristic when OpenRouter is active.

1 · Architecture & module boundaries

Introduce new module js/providers.js exporting two provider objects that share one interface:

provider = {
  id,                       // 'anthropic' | 'openrouter'
  label,
  buildRequest(payload),    // -> { url, headers, body }   translate neutral payload to wire format
  parseResponse(json),      // -> { text, usage }          normalize response shape
  estimateInputTokens(payload, apiKey, signal),  // count_tokens (anthropic) | heuristic (openrouter)
  models,                   // curated [{ id, label, inputPrice, outputPrice }]
  keyStorageKey,            // 'anthropic_key' | 'openrouter_key'
}

chat.js keeps owning conversation state, UI wiring, history, and cost-preview orchestration, but delegates the three provider-specific things through this interface. buildPayload / buildSystemBlocks stay provider-neutral — they already emit a clean { model, system, messages } shape; each provider translates that into its own wire format inside buildRequest.

Why a separate module: isolates the divergent wire format behind a testable seam, keeps the already-large chat.js from sprawling, matches the repo's small single-purpose module pattern.

2 · The two providers' wire formats

Anthropic (unchanged behavior)

  • POST /v1/messages; headers x-api-key, anthropic-version, anthropic-dangerous-direct-browser-access.
  • Body passes the neutral payload through as-is (already Anthropic-shaped: system blocks with cache_control, content[] messages).
  • parse: content[].filter(type==='text') join; usage maps input / cache_creation / cache_read / output.
  • estimate: real count_tokens call (accurate).

OpenRouter (neutral → OpenAI chat)

  • POST https://openrouter.ai/api/v1/chat/completions; header Authorization: Bearer <key> (+ optional HTTP-Referer / X-Title). No browser-access header — permissive CORS.
  • system blocks → flattened into one {role:'system'} message.
  • messages content[] → joined to plain strings, so stored Anthropic-style history converts at send time → switching providers mid-chat works.
  • parse: choices[0].message.content; usage maps prompt_tokens→in, completion_tokens→out.
  • estimate: chars/4 × model inputPrice heuristic, no API call.
YAGNI — prompt caching dropped for OpenRouter. No cache_control is sent; it's model-dependent there and adds real complexity. The heuristic treats all input as uncached. Cost preview branches: Anthropic keeps the cache-write / cache-read two-number display; OpenRouter shows a single flat estimate.

3 · UI, state & key storage

Provider selector

New <select> in the chat controls (index.html, beside the model select): Anthropic (direct) / OpenRouter. Persisted to new key /fintool/provider via the storage.js wrapper. Default anthropic → existing users see no change.

Two API keys, not one

Keep anthropic_key; add openrouter_key. The single #chat-key input rebinds to the active provider — switching the dropdown swaps the input's value and its "✓ key saved locally" state. Each key persists independently.

Provider-driven model dropdown

On provider change, repopulate #chat-model from the active provider's curated models and restore that provider's last-used model. OpenRouter additionally gets a free-text slug override (e.g. openai/gpt-5, google/gemini-2.5-pro); non-empty override wins over the dropdown. Hidden when Anthropic is active.

Switching provider mid-conversation

Reuses the existing noteContextChange() path — drops an italic "Provider changed to X — subsequent answers use {model}" notice, preserves history, invalidates in-flight sends. No new mechanism.

Pricing

Today's INPUT_PRICE_PER_1M is Anthropic-only. Each provider carries its own curated prices inline in its models list; the Anthropic provider keeps using the existing constant. A free-text OpenRouter slug has no known price → preview falls back to "tokens only, price unavailable".

4 · Testing

Vitest already covers parsers + importer. Add a new providers suite that needs no network:

  • Neutral → Anthropic: buildRequest preserves system blocks, cache_control, and content[] messages; correct URL + headers.
  • Neutral → OpenRouter: system blocks flatten to one system message; content[] history converts to string content; bearer header; correct URL.
  • parseResponse round-trip: a sample Anthropic content[] response and a sample OpenRouter choices[] response both normalize to the same { text, usage } shape.
  • Heuristic estimate: chars/4 × price math, and the price-unavailable fallback for an unknown slug.
  • Provider switch mid-history: a conversation built under Anthropic re-serializes cleanly for an OpenRouter send (the convert-at-send-time guarantee).

No linter in repo; keep the console.assert row-count guards in app.js:main as-is.

Open questions / deferred (YAGNI)

  • Prompt caching for OpenRouter Anthropic models — deliberately deferred.
  • Live /models fetch & live pricing — deferred (curated list chosen).
